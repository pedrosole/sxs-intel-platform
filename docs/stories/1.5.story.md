# Story 1.5 — Auto-Extract Client Summaries from Agent Outputs

**Status:** Done
**Tipo:** Enhancement
**Prioridade:** MEDIUM
**Estimativa:** P (pequena)

---

## Descricao

A tabela `client_summaries` tem campos `brand_voice_summary` e `positioning_summary` que deveriam ser preenchidos automaticamente a partir dos outputs de @debora e @athena. Atualmente, ambos ficam NULL mesmo apos 2 execucoes completas.

**Evidencia:** Client summary da Gabriela Lopes mostra:
```json
{
  "brand_voice_summary": null,
  "positioning_summary": null,
  "total_jobs": 2
}
```

Esses resumos sao fundamentais para o pipeline RECURRING — quando o cliente volta no mes seguinte, @debora e @athena precisam do contexto anterior para nao repetir trabalho.

---

## Escopo

### IN
- Na funcao `updateClientSummary` em `operations.ts` (ou no orchestrator apos pipeline):
  - Extrair brand_voice_summary do output da @debora (secao "Brand Voice" — tom, vocabulario, estilo)
  - Extrair positioning_summary do output da @athena (secao "Narrativa Central" — mensagem-mae, promessa, diferenciais)
  - Salvar na `client_summaries` junto com o update existente
- Logica de extracao:
  - Buscar secoes por headers Markdown (`### 3. Brand Voice`, `### 1. Narrativa Central`)
  - Extrair texto entre o header e o proximo header de mesmo nivel
  - Truncar a 1000 chars se necessario (resumo, nao output completo)

### OUT
- Extracao de outros campos (avg_qa_score, key_decisions) — futuro
- Mudancas no schema (campos ja existem)

---

## Criterios de Aceitacao

- [x] AC1: Apos pipeline completo, `brand_voice_summary` contem resumo extraido do output @debora (nao null)
- [x] AC2: Apos pipeline completo, `positioning_summary` contem resumo extraido do output @athena (nao null)
- [x] AC3: Se extracao falhar (formato inesperado), campo fica null sem quebrar pipeline
- [x] AC4: Em execucoes RECURRING, os resumos anteriores sao carregados no contexto dos agentes (ja implementado no orchestrator — validar que funciona com dados reais)
- [x] AC5: Build passa sem erros
